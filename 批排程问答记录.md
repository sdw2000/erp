# 💡 用户问题与解答记录

## 用户问题

> **"批排程时，没进入排程的数量存放在哪里？"**

例如：订单100卷，用户只排60卷，剩余40卷应该如何管理？

---

## 完整答案

### 一句话答案
```
未排程数量存放在 sales_order_items.pending_qty 字段中，
每次排程后自动减少，用户下次查询时看到的是新的待排数量
```

### 核心三个字段

| 字段 | 表名 | 含义 | 初值 | 变化 | 完成 |
|------|------|------|------|------|------|
| `pending_qty` | sales_order_items | **未排程数量** | order_qty | 每次排程减少 | = 0 |
| `schedule_qty` | schedule_order_item | 本次排程数量 | NULL | 排程时记录 | 历史记录 |
| `order_qty` | sales_order_items | 订单原始数量 | 订单创建 | 不变 | 参考值 |

---

## 完整数据流

```
【第1天】订单创建
  PO-001: order_qty=100, pending_qty=100

【第2天】用户排程60卷
  输入：schedule_qty = 60
  验证：60 ≤ 100 ✓
  后端：
    ✓ 创建 schedule_order_item (schedule_qty=60)
    ✓ 更新 pending_qty = 100 - 60 = 40
    ✓ 创建工序任务

【第3天】查看待排程订单
  显示：PO-001 的 pending_qty = 40

【第5天】继续排程40卷中的30卷
  输入：schedule_qty = 30
  验证：30 ≤ 40 ✓
  后端：
    ✓ 创建 schedule_order_item (schedule_qty=30)
    ✓ 更新 pending_qty = 40 - 30 = 10

【第7天】最后排程10卷
  输入：schedule_qty = 10
  后端：
    ✓ pending_qty = 10 - 10 = 0 ✓ 完全排程

【最终统计】
  排程历史（schedule_order_item）：
    排程1：60卷
    排程2：30卷
    排程3：10卷
    总计：100卷 ✓

  pending_qty = 0 表示完全排程 ✓
```

---

## 为什么这样设计？

### ✅ 优势

1. **清晰**
   - pending_qty 直观显示"还能排多少"
   - 用户不需要心算：100-60-30-...

2. **可追溯**
   - schedule_order_item 记录每次排程的数量
   - 知道"排过几次"、"每次多少"

3. **支持分批**
   - 可以多次从 pending_qty 中选择排程
   - 灵活应对"先排急单，后排普通"的业务

4. **实时同步**
   - 前端排程后重新加载
   - 自动显示新的 pending_qty
   - 不需要手动维护

5. **易于验证**
   - schedule_qty ≤ pending_qty 是明确的约束
   - 简单有效的数据一致性保证

### 🔄 数据流转

```
订单创建
  ↓
pending_qty = order_qty (初值100)
  ↓
【用户排程】
  ↓
schedule_qty = 60 (用户输入)
  ↓
【后端处理】
  ├─ 记录：schedule_order_item.schedule_qty = 60
  └─ 更新：pending_qty = 40
  ↓
【前端重新加载】
  ↓
显示：pending_qty = 40（新的待排数量）
  ↓
【用户可继续排程】
  ↓
从40中选择排程...（循环）
```

## 实现关键点

### 前端

```javascript
// 1. 表格列
<el-table-column prop="pending_qty" label="待排数量" />
<el-table-column prop="schedule_qty" label="本次排程">
  <el-input-number 
    v-model="row.schedule_qty" 
    :max="row.pending_qty"  // ← 关键验证
  />
</el-table-column>

// 2. 加载订单时初始化
this.pendingOrders = orders.map(o => ({
  ...o,
  schedule_qty: o.pending_qty  // 默认 = 待排数量
}))

// 3. 提交排程时收集
const details = selectedOrders.map(o => ({
  order_item_id: o.id,
  schedule_qty: o.schedule_qty  // 用户输入的值
}))

// 4. 排程后重新加载
this.loadPendingOrders()  // pending_qty 会自动更新
```

### 后端

```sql
-- 排程时的关键SQL
UPDATE sales_order_items
SET pending_qty = pending_qty - ?,  -- schedule_qty
    updated_at = NOW()
WHERE id = ?
  AND pending_qty >= ?;  -- 验证不会为负
```

## 常见问题

### Q1: 如果用户多次查看，pending_qty 会不会重复减少？
**A**: 不会。因为：
- pending_qty 只在排程提交时更新**一次**
- 下次加载待排程列表时显示的是**新的pending_qty**
- 用户看到的总是当前的、正确的待排数量

### Q2: 如何验证数据一致性？
**A**: 使用定期检查SQL：
```sql
SELECT order_no, 
  CASE WHEN (pending_qty + produced_qty) <= order_qty 
    THEN '✓' ELSE '❌ 错误' END AS status
FROM sales_order_items
```

### Q3: pending_qty 能为负数吗？
**A**: 不能，通过：
- 数据库约束：`CHECK (pending_qty >= 0)`
- 后端验证：`schedule_qty ≤ pending_qty`
- 前端验证：input-number 的 `:max="row.pending_qty"`

### Q4: 如果排程中途取消，pending_qty 怎么恢复？
**A**: 取消排程时添加：
```sql
UPDATE sales_order_items
SET pending_qty = pending_qty + cancelled_qty  -- 恢复
WHERE id = order_item_id
```

## 总结对比

### 旧方式（不推荐）
```
❌ pending_qty 不变，用户需要心算
❌ 无法追踪分批排程的历史
❌ 用户容易输入超过剩余的数量
❌ 后续查询时需要减去所有历史排程
```

### 新方式（推荐）✓
```
✓ pending_qty 自动更新，显示真实待排数量
✓ schedule_order_item 记录每次排程历史
✓ 验证 schedule_qty ≤ pending_qty 防止错误
✓ 下次查询时直接看新的 pending_qty，无需计算
✓ 支持分批排程，灵活高效
```
