## 📋 关键决策点详解

### 决策1：库存检查（第189-220行）

```
┌─ 有库存充足？
│  ├─ YES → canRewindItems（可直接复卷）
│  └─ NO → needCoatingItems（需要涂布）
└─
```

**核心SQL：**
```sql
-- 检查是否有匹配的库存
SELECT * FROM tape_stock
WHERE color_code = ?        -- 必须颜色相同
  AND thickness = ?         -- 必须厚度相同
  AND width >= ?            -- 宽度要足够（≥订单宽度）
  AND total_rolls >= ?      -- 库存数量要足够
  AND status = 'available'  -- 必须是可用状态
LIMIT 1;

-- 如果查询结果为空，订单进入涂布
```

---

### 决策2：订单分组（第227-232行）

```
needCoatingItems (所有需要涂布的订单)
  │
  ├─ 按"颜色+厚度"分组
  │
  ├─ "蓝色_100" → [PO-001, PO-003, PO-005, ...]
  ├─ "蓝色_150" → [PO-002, PO-004, ...]
  ├─ "绿色_120" → [PO-006, PO-007, ...]
  └─ "红色_100" → [PO-008, ...]
  
  ✅ 优势：相同规格的订单合并涂布，提高效率
```

**分组代码：**
```java
Map<String, List<Map<String, Object>>> groupedCoatingItems = 
    needCoatingItems.stream()
        .collect(Collectors.groupingBy(item -> {
            String colorCode = item.get("color_code").toString();
            String thickness = item.get("thickness").toString();
            return colorCode + "_" + thickness;  // ← 分组key
        }));
```

## 🔧 常见的流程变体

### 变体1：有部分库存的情况

```
订单1：需要100卷库存
库存：只有50卷

【结果】
→ 订单进入涂布（因为库存不足）
→ 或者可以修改逻辑支持"分批"

当前代码：要么全部用库存，要么全部涂布（不支持混合）
```

### 变体2：多种规格库存的情况

```
订单：蓝色 + 100μm + 1200mm宽

库存选项1：蓝色 + 100μm + 1200mm宽 + 100卷 ✅ 匹配
库存选项2：蓝色 + 100μm + 1500mm宽 + 200卷 ✅ 也可以（宽度充足）
库存选项3：蓝色 + 100μm + 1000mm宽 + 300卷 ❌ 不匹配（宽度不足）

【系统选择】
选择第一个满足条件的库存（第195-207行的循环）
```

### 变体3：跨天排程的情况

```
【当前逻辑】
一次排程只生成一个涂布任务

【改进方向】
可以改为：按机器容量分割任务
    大涂布任务(2天)
    → 分割为两个1天的任务
```

## ✅ 总结

涂布排程的核心逻辑：

```
1️⃣ 查询待排程订单
         ↓
2️⃣ 检查库存是否充足
         ↓
3️⃣ 库存不足 → 进入涂布流程
         ↓
4️⃣ 按颜色+厚度分组
         ↓
5️⃣ 为每组创建涂布任务
         ↓
6️⃣ 分配设备（轮询）
         ↓
7️⃣ 设置工艺参数
         ↓
✅ 订单进入"待涂布"状态
```

**关键点：** 是否进入涂布取决于库存！
- 有足够库存 → 复卷
- 无足够库存 → 涂布
