## 🔑 关键概念

### 库存检查逻辑

```java
// 代码位置：ProductionScheduleServiceImpl.java 第189-220行

for (订单 : 待排程订单) {
  // 查询是否有匹配的库存
  库存 = 查询库存(
    color_code = 订单.color_code,      // 颜色必须相同
    thickness = 订单.thickness,        // 厚度必须相同
    width >= 订单.width,               // 宽度足够即可
    total_rolls >= 订单.pending_qty,   // 数量要足够
    status = 'available'               // 必须可用
  )
  
  if (库存存在) {
    订单 → 复卷流程
  } else {
    订单 → 涂布流程 ← 您的情况
  }
}
```

### 为什么这样设计？

| 设计目标 | 实现方式 | 优点 |
|---------|--------|------|
| **成本优化** | 优先用库存 | 减少涂布成本 |
| **效率提升** | 并行处理 | 库存直接复卷 |
| **自动化** | 无需人工判断 | 减少错误 |
| **分组优化** | 相同规格合并 | 提高设备利用率 |

## ✅ 诊断流程

```
【问题】所有订单都进入待涂布

【第1步】检查库存
  SELECT COUNT(*) FROM tape_stock;
  
  ├─ 结果=0 → ❌ 库存为空，这就是原因！
  ├─ 结果>0 → 继续第2步
  
【第2步】检查库存规格是否与订单匹配
  SELECT color_code, thickness, width FROM tape_stock;
  SELECT color_code, thickness, width FROM sales_order_items;
  
  ├─ 规格不匹配 → ❌ 添加匹配的库存
  ├─ 规格匹配 → 继续第3步
  
【第3步】检查库存数量是否足够
  SELECT total_rolls FROM tape_stock;
  SELECT pending_qty FROM sales_order_items;
  
  ├─ 数量不足 → ❌ 增加库存数量
  ├─ 数量足够 → 继续第4步
  
【第4步】检查库存状态
  SELECT status FROM tape_stock;
  
  ├─ status ≠ 'available' → ❌ 修改状态
  ├─ status = 'available' → 继续第5步
  
【第5步】检查涂布设备是否存在
  SELECT * FROM equipment WHERE type = 'COATING';
  
  ├─ 设备存在 → ✅ 系统正常工作
  ├─ 设备不存在 → ❌ 添加涂布设备
  
【结论】
排程时的决策：
├─ 如果通过所有库存检查 → 订单进入复卷 ✅
└─ 如果任何检查失败 → 订单进入涂布 ⚠️
```

## 📌 重要提醒

### ⚠️ 不要忘记

1. **库存必须存在** - 否则所有订单都会进入涂布
2. **规格必须匹配** - 颜色、厚度、宽度都要对
3. **设备必须存在** - 涂布机、复卷机等设备要提前创建
4. **状态必须正确** - 库存状态必须是 `available`

### ✅ 最佳实践

1. **定期检查库存** - 确保库存表有数据
2. **标准化规格代码** - 颜色、厚度等要统一格式
3. **提前配置设备** - 在排程前把所有设备配置好
4. **保存审计日志** - 记录每次排程的决策过程

## ❓ 常见问题

**Q: 所有订单进入涂布是bug吗？**
A: 不是。这是正常的设计行为 - 当库存不足时，订单需要通过涂布来补充库存。

**Q: 如何强制所有订单都进入复卷？**
A: 需要提前在库存表中准备足够的库存。

**Q: 如何强制所有订单都进入涂布？**
A: 清空库存表即可（但这不是推荐做法）。

**Q: 可以改变分组规则吗？**
A: 可以，但需要修改 `ProductionScheduleServiceImpl.java` 第227-232行。

**Q: 涂布完成后会自动进入复卷吗？**
A: 不会。涂布完成后，生成的母卷会进入库存表，之后的订单才能用该库存。
