# 🎯 批排程数量逻辑完整设计

## 📌 核心问题

当用户批量排程时，比如：
- 订单 PO-001 待排 100卷，用户只排 60卷
- 剩余 40卷应该存放在哪里？
- 后续应该如何处理这 40卷？

---

## 💾 数据存放位置分析

### 1. 关键表结构

```
【销售订单明细表】sales_order_items
┌────────────────────────────────────────┐
│ id (主键)                              │
│ order_id                               │
│ material_code                          │
│ order_qty          ← 订单总数量        │
│ produced_qty       ← 已生产数量        │
│ pending_qty        ← 待排数量          │
│ status = 'pending' ← 订单状态          │
└────────────────────────────────────────┘

【排程-订单关联表】schedule_order_item
┌────────────────────────────────────────┐
│ id (主键)                              │
│ schedule_id                            │
│ order_item_id      ← 关联销售订单明细  │
│ schedule_qty       ← 本次排程数量      │
│ status = 'pending' ← 排程状态          │
└────────────────────────────────────────┘
```

### 2. 未排程数量的存放位置

```
未排程数量 = pending_qty - schedule_qty

【存放位置】两个地方同时保存：

① sales_order_items.pending_qty
   ├─ 初始值 = order_qty
   ├─ 每次排程后 = pending_qty - schedule_qty
   └─ 用途：查询订单还有多少未排程

② schedule_order_item.schedule_qty  
   ├─ 记录本次排程数量
   └─ 用途：追溯排程历史

【示例流程】
初始状态：
  pending_qty = 100卷

第1次排程：
  schedule_qty = 60卷
  → pending_qty 更新为 40卷
  → schedule_order_item 记录 schedule_qty=60

第2次排程：
  schedule_qty = 30卷 (从剩余40卷中选)
  → pending_qty 更新为 10卷
  → schedule_order_item 记录 schedule_qty=30

最终：
  pending_qty = 10卷 (最后一次未排程数量)
```

## 🗄️ 数据库实现

### SQL示例

```sql
-- 1. 初始化：创建销售订单
INSERT INTO sales_order_items 
(order_id, material_code, order_qty, produced_qty, pending_qty, status)
VALUES 
(1, 'TAPE-001', 100, 0, 100, 'pending');

-- 2. 用户排程时：创建排程关联
INSERT INTO schedule_order_item 
(schedule_id, order_item_id, schedule_qty, status)
VALUES 
(1, 1, 60, 'pending');

-- 3. 更新待排数量
UPDATE sales_order_items 
SET pending_qty = pending_qty - 60
WHERE id = 1;
-- 结果：pending_qty = 40

-- 4. 查询待排程订单（显示还有40卷未排）
SELECT 
  soi.id,
  so.order_no,
  soi.material_code,
  soi.order_qty,           -- 订单原始数量：100
  soi.produced_qty,        -- 已生产：0
  soi.pending_qty          -- 待排：40（不是100）
FROM sales_order_items soi
LEFT JOIN sales_orders so ON soi.order_id = so.id
WHERE soi.pending_qty > 0;

-- 5. 查询排程历史（知道排过多少次）
SELECT 
  s.schedule_no,
  soi.order_item_id,
  soi.schedule_qty,
  soi.status
FROM schedule_order_item soi
LEFT JOIN production_schedule s ON soi.schedule_id = s.id
WHERE soi.order_item_id = 1
ORDER BY s.created_at DESC;
```

## 🎯 核心数据流

### 待排数量的更新逻辑

```
【四个关键时间点】

① 订单创建时：
   pending_qty = order_qty (初始值)

② 排程时：
   pending_qty 减去 schedule_qty
   pending_qty = pending_qty - schedule_qty
   
③ 生产完成时：
   produced_qty 增加（可选：pending_qty 也减少，或只在排程时减）
   
④ 订单完成时：
   pending_qty = 0
   status = 'completed'

【两种更新策略的对比】

策略A：在排程时减少 pending_qty（推荐）
  优点：
  ├─ 排程 = 已承诺要生产的
  ├─ pending_qty 代表真正还未分配的
  └─ 逻辑清晰
  
  流程：
  pending_qty → (排程) → pending_qty - schedule_qty
             → (生产) → produced_qty + output_qty

策略B：在生产完成时减少 pending_qty
  优点：
  ├─ 只统计实际完成的
  └─ 更保守
  
  流程：
  pending_qty 不变
             → (排程) → schedule_qty 记录
             → (生产) → produced_qty + output_qty, pending_qty 减少
```

```sql
SELECT 
  soi.id,

  so.order_no,
  so.customer,

  soi.color_code
FROM sales_order_items soi
LEFT JOIN sales_orders so ON soi.order_id = so.id

  AND soi.status = 'pending'
ORDER BY so.customer_level DESC, soi.delivery_date ASC;
```

### 2. 更新待排数量（排程后）

```sql
-- 方案1：直接减法
UPDATE sales_order_items
SET pending_qty = pending_qty - ?,    -- ? = 用户输入的 schedule_qty
    updated_at = NOW()
WHERE id = ?;

-- 方案2：使用触发器自动更新
DELIMITER $$
CREATE TRIGGER update_pending_qty_after_schedule
AFTER INSERT ON schedule_order_item
FOR EACH ROW
BEGIN
  UPDATE sales_order_items
  SET pending_qty = pending_qty - NEW.schedule_qty
  WHERE id = NEW.order_item_id;
END $$
DELIMITER ;
```

### 3. 验证排程数量不超过待排

```sql
-- 前端验证：schedule_qty ≤ pending_qty

-- 后端验证：
SELECT pending_qty FROM sales_order_items WHERE id = ?;
-- 检查：输入的 schedule_qty ≤ 查询结果的 pending_qty
```

```

  pending_qty: 10 → 0

  ✓ 订单完全排程
  ✓ 有3条排程记录

```

```

  ✓ 创建3条 schedule_order_item
```

## 🛡️ 数据一致性保证

### 约束条件

```
1. pending_qty 永远不能为负数
   ├─ SQL约束：pending_qty >= 0
   └─ 后端验证：schedule_qty ≤ pending_qty

2. produced_qty ≤ order_qty
   ├─ 报工时检查
   └─ 不能生产超过订单数量

3. produced_qty ≤ order_qty - pending_qty
   ├─ 只有已排程的才能生产
   └─ pending_qty 保护未排程部分

4. order_qty = produced_qty + pending_qty + 其他(待排未生产)
   ├─ 三者之和必须等于订单量
   └─ 定期检查数据一致性
```

### SQL约束

```sql
ALTER TABLE sales_order_items 
ADD CONSTRAINT check_pending_qty CHECK (pending_qty >= 0);

ALTER TABLE sales_order_items 
ADD CONSTRAINT check_produced_qty CHECK (produced_qty <= order_qty);

ALTER TABLE sales_order_items 
ADD CONSTRAINT check_quantities 
CHECK (produced_qty + pending_qty <= order_qty);
```

## 🔄 前后端交互

### 前端逻辑

```javascript
// data 中添加
data() {
  return {
    pendingOrders: [],  // 待排程订单列表
    selectedOrders: []  // 用户选中的订单
  }
}

// 加载待排程订单
loadPendingOrders() {
  const res = await getPendingOrderItems(this.pendingParams)
  // 后端返回的已经是 pending_qty > 0 的订单
  this.pendingOrders = res.data.map(order => ({
    ...order,
    schedule_qty: order.pending_qty  // 默认 = 待排数量
  }))
}

// 用户修改排程数量时验证
validateScheduleQty(row) {
  if (row.schedule_qty > row.pending_qty) {
    this.$message.warning(`最多只能排${row.pending_qty}卷`)
    row.schedule_qty = row.pending_qty
  }
  if (row.schedule_qty < 0) {
    row.schedule_qty = 0
  }
}

// 提交排程
async batchSchedule() {
  const details = this.selectedOrders.map(order => ({
    order_item_id: order.id,
    schedule_qty: order.schedule_qty  // 用户输入的值
  }))
  
  const res = await autoSchedule({
    scheduleDate: this.autoScheduleForm.scheduleDate,
    details: details
  })
  
  // 排程成功后重新加载待排程列表
  if (res.code === 0) {
    this.loadPendingOrders()  // pending_qty 会自动更新
  }
}
```

### 后端API

```javascript
// POST /api/production/schedule/auto
// request body:
{
  scheduleDate: "2026-01-10",
  details: [
    { order_item_id: 1, schedule_qty: 60 },
    { order_item_id: 2, schedule_qty: 50 }
  ]
}

// 后端处理逻辑：
1. 验证每个 schedule_qty ≤ pending_qty
2. 开始事务
3. 对每个订单：
   a. 创建 schedule_order_item 记录
   b. 更新 sales_order_items.pending_qty -= schedule_qty
   c. 创建工序任务（涂布、复卷等）
4. 提交事务
5. 返回结果

// response:
{
  code: 0,
  message: "排程成功",
  data: {
    scheduleNo: "PS-20260110-001",
    itemCount: 2,
    totalQty: 110
  }
}
```

## ✅ 总结

| 要素 | 说明 |
|------|------|
| **未排程数量存放** | `sales_order_items.pending_qty` |
| **排程历史记录** | `schedule_order_item.schedule_qty` |
| **初始值** | `pending_qty = order_qty` |
| **排程后更新** | `pending_qty = pending_qty - schedule_qty` |
| **完全排程条件** | `pending_qty = 0` |
| **分批排程** | 支持多次排程，每次从 `pending_qty` 中选择 |
| **数据一致性** | 通过约束和触发器保证 |
