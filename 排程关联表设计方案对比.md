# 🎯 排程关联表设计方案对比 - 两种方案深度分析

## 核心问题
> 排程记录应该存在哪里？
> - 选项A: 直接扩展订单表？
> - 选项B: 创建独立的排程关联表？

---

## 📊 方案对比

### 方案A ✅ 【推荐】创建独立排程关联表 (schedule_order_item)

**核心思想**：订单信息和排程信息分离

#### 数据库设计
```sql
-- 销售订单表（保持简洁）
CREATE TABLE sales_orders (
  id INT PRIMARY KEY,
  order_no VARCHAR(50),
  customer VARCHAR(100),
  delivery_date DATE,
  order_qty INT,          -- 订单总数量
  status VARCHAR(20),
  created_at TIMESTAMP
);

-- 销售订单明细表（产品级别）
CREATE TABLE sales_order_items (
  id INT PRIMARY KEY,
  order_id INT,           -- 关联销售订单
  material_code VARCHAR(50),
  item_qty INT,           -- 本明细的订单数量
  pending_qty INT,        -- 本明细的待排数量 ← 关键字段
  produced_qty INT,       -- 本明细的已生产数量
  FOREIGN KEY (order_id) REFERENCES sales_orders(id)
);

-- 【新增】排程记录表（排程关联表）
CREATE TABLE schedule_order_item (
  id INT PRIMARY KEY,
  order_item_id INT,      -- 关联sales_order_items
  schedule_date DATE,     -- 排程日期
  schedule_qty INT,       -- 本次排程数量
  status VARCHAR(20),     -- 排程状态
  created_at TIMESTAMP,
  FOREIGN KEY (order_item_id) REFERENCES sales_order_items(id)
);

-- 各工序任务表（工序层级）
CREATE TABLE schedule_task (
  id INT PRIMARY KEY,
  schedule_item_id INT,   -- 关联schedule_order_item
  task_type VARCHAR(20),  -- 涂布、复卷等
  quantity INT,
  status VARCHAR(20),
  FOREIGN KEY (schedule_item_id) REFERENCES schedule_order_item(id)
);
```

#### 业务流程
```
【订单创建】
  sales_orders.order_qty = 100
    ↓
【明细初始化】
  sales_order_items.item_qty = 100
  sales_order_items.pending_qty = 100
  sales_order_items.produced_qty = 0
    ↓
【查询待排程】WHERE pending_qty > 0
    ↓
【用户排程60卷】
  INSERT INTO schedule_order_item
  (order_item_id=1, schedule_qty=60, ...)
    ↓
【更新明细】
  UPDATE sales_order_items
  SET pending_qty = pending_qty - 60  (100→40)
  WHERE id = 1
    ↓
【创建工序任务】
  INSERT INTO schedule_task
  (schedule_item_id=1, task_type='涂布', quantity=60)
    ↓
【用户继续排程30卷】
  INSERT INTO schedule_order_item
  (order_item_id=1, schedule_qty=30, ...)
    ↓
【再次更新明细】
  UPDATE sales_order_items
  SET pending_qty = pending_qty - 30  (40→10)
  WHERE id = 1
    ↓
【查询排程历史】
  SELECT * FROM schedule_order_item
  WHERE order_item_id = 1
  -- 显示两条排程记录：60卷 + 30卷 = 90卷已排程
```

#### 优点 ✅
| 优点 | 说明 |
|------|------|
| **数据分离** | 订单数据和排程数据分离，职责清晰 |
| **排程溯源** | 每条排程记录完整保存，可查询排程历史 |
| **分批排程** | 天然支持同一订单多次排程，记录完整 |
| **灵活扩展** | 排程表可添加更多字段（计划人、优先级等）无需改订单表 |
| **并发安全** | 排程表独立，减少并发冲突 |
| **报表分析** | 可单独统计排程数据，用于分析 |

#### 缺点 ❌
| 缺点 | 说明 |
|------|------|
| **多表关联** | 查询需JOIN多个表，略微增加复杂度 |
| **代码量** | 需要维护额外的schedule_order_item表逻辑 |

#### 适用场景 🎯
```
✅ 需要追踪排程历史
✅ 同一订单多次排程
✅ 需要排程数据分析
✅ 生产环节复杂，需要按排程单组织任务
✅ 大规模生产（MES必需）
```

### 方案B ❌ 直接扩展订单表

**核心思想**：订单表和排程信息合并

#### 数据库设计
```sql
-- 方式B1：在销售订单表中添加字段
CREATE TABLE sales_orders (
  id INT PRIMARY KEY,
  order_no VARCHAR(50),
  customer VARCHAR(100),
  order_qty INT,          -- 订单总数量
  pending_qty INT,        -- 待排程数量 ← 在订单表中
  produced_qty INT,       -- 已生产数量
  scheduled_qty INT,      -- 已排程数量（可选）
  created_at TIMESTAMP
);

-- 方式B2：在订单明细表中添加字段
CREATE TABLE sales_order_items (
  id INT PRIMARY KEY,
  order_id INT,
  material_code VARCHAR(50),
  item_qty INT,
  pending_qty INT,        -- 待排程数量 ← 在明细表中
  produced_qty INT,
  created_at TIMESTAMP
);
```

#### 业务流程
```
【订单创建】
  sales_order_items.item_qty = 100
  sales_order_items.pending_qty = 100
    ↓
【用户排程60卷】
  UPDATE sales_order_items
  SET pending_qty = 40
  WHERE id = 1
    ↓
【更新销售订单表】
  UPDATE sales_orders
  SET pending_qty = 40 (从明细表汇总)
    ↓
【用户继续排程30卷】
  UPDATE sales_order_items
  SET pending_qty = 10
  WHERE id = 1
    ↓
【问题1】排程历史在哪里？
  无法查询曾经排过60卷和30卷
  
【问题2】如何追踪排程？
  需要额外的排程日志表才能记录
```

#### 优点 ✅
| 优点 | 说明 |
|------|------|
| **结构简洁** | 字段少，表少，概念清晰 |
| **查询速度** | 不需多表JOIN，查询快 |
| **实现快** | 初期开发快，代码简单 |

#### 缺点 ❌
| 缺点 | 说明 |
|------|------|
| **无排程历史** | 只保存当前状态，无法查询排程记录 |
| **分批排程困难** | pending_qty只记录剩余数，无法知道已排过多少次 |
| **工序任务关联弱** | 无排程单，工序任务与排程的关系模糊 |
| **扩展性差** | 日后要添加排程信息就很困难 |
| **业务数据混乱** | 订单表承载太多职责，数据库设计不规范 |

#### 不适用场景 ❌
```
❌ 需要排程审计
❌ 同订单多次排程
❌ 需要排程数据分析
❌ 生产环节复杂
❌ MES系统
```

## 🏆 推荐方案

### ✅ 使用方案A：创建独立排程关联表

**理由**：
1. **MES系统的标准做法** - 所有大型MES系统都采用这种设计
2. **完全支持分批排程** - 同一订单可多次排程，每条记录清晰
3. **排程溯源** - 符合生产管理的审计要求
4. **未来扩展** - 排程表可添加优先级、计划人等信息

**vs 方案B的差别**：
- 方案B只能记录`最终状态`（待排数量），无法记录`排程过程`
- 如果后续需要"分析今天排了几批"就很困难了
- MES的价值就在于可追溯，方案B失去了这个优势

## 📝 快速参考

### 关键表说明

| 表名 | 作用 | 关键字段 |
|------|------|--------|
| **sales_orders** | 销售订单头 | order_no, order_qty |
| **sales_order_items** | 销售订单明细 | material_code, item_qty, **pending_qty**, produced_qty |
| **schedule_order_item** | 排程关联表 | order_item_id, **schedule_qty**, schedule_date |
| **schedule_task** | 工序任务 | schedule_item_id, task_type, quantity |

### 关键流程

| 场景 | SQL操作 |
|------|--------|
| **创建订单明细** | `INSERT sales_order_items: item_qty=100, pending_qty=100` |
| **查询待排程** | `SELECT * FROM sales_order_items WHERE pending_qty > 0` |
| **排程60卷** | `INSERT schedule_order_item (60 卷)` + `UPDATE sales_order_items: pending_qty -= 60` |
| **查询排程历史** | `SELECT * FROM schedule_order_item WHERE order_item_id = 1` |
| **验证排程** | `schedule_qty ≤ pending_qty` |
